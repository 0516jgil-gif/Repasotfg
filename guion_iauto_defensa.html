Portada — IAuto

⏱ ~20 segDiapositiva 1 de 16

Buenos días. Somos Miguel Barbero y Julio Gil y vamos a presentar IAuto, una aplicación web orientada a digitalizar la experiencia de compra y gestión de vehículos de segunda mano.

La idea principal no es crear únicamente un escaparate de coches. Hemos desarrollado una plataforma con distintos perfiles de usuario, operaciones de gestión y un asistente virtual conectado al catálogo real.

3

Contexto y Justificación

⏱ ~60 segDiapositiva 3 de 16

El sector de la automoción también se está trasladando al entorno digital. Antes de desplazarse a un concesionario, el cliente quiere consultar modelos, precios y disponibilidad desde casa.

Sin embargo, consultar información no siempre es suficiente. Cuando el usuario tiene dudas concretas —por ejemplo qué vehículo encaja en un presupuesto o si todavía queda stock— espera una respuesta rápida.

IAuto nace para cubrir esa necesidad: agrupa el catálogo, la relación con el cliente y la operativa interna en una misma aplicación web.

💡

Ejemplo: Un visitante puede consultar el inventario sin registrarse. Si encuentra un coche interesante, puede entrar en su ficha y revisar precio, características e imágenes antes de dar el siguiente paso.

➡️

Transición: A partir de ese problema definimos tres objetivos principales.

4

Objetivos del Proyecto

⏱ ~55 segDiapositiva 4 de 16

El primer objetivo era construir una aplicación web accesible desde el navegador, evitando que el usuario dependa de una aplicación instalada.

El segundo era ofrecer una interfaz intuitiva. Por eso el catálogo permite buscar por marca o modelo y las acciones principales —como ver detalles, guardar favoritos o iniciar una compra— aparecen de forma directa.

El tercer objetivo era integrar inteligencia artificial con una utilidad concreta dentro del negocio: ayudar al cliente a resolver dudas y elegir un vehículo.

💡

Ejemplo: Si un cliente busca un coche familiar, puede filtrar el catálogo y después preguntar al asistente qué opciones disponibles encajan mejor con su presupuesto.

➡️

Análisis de Mercado

⏱ ~60 segDiapositiva 5 de 16

Al analizar el mercado tomamos como referencia plataformas conocidas como Coches.net y Flexicar.

Coches.net destaca por la variedad de anuncios y facilita la búsqueda inicial. Flexicar, por su parte, tiene una presencia importante en vehículos de segunda mano.

Nuestra propuesta no busca competir en volumen dentro de un TFG, sino demostrar una experiencia integrada: catálogo, gestión de usuarios, favoritos, solicitudes de compra, seguimiento interno y atención inmediata mediante IA.

💡

6

Tecnologías Utilizadas

⏱ ~2 minDiapositiva 6 de 16

El lenguaje principal del proyecto es JavaScript. Lo utilizamos tanto en el navegador como en la lógica del servidor. Con JavaScript controlamos los formularios, las búsquedas del catálogo, el cálculo del precio de una compra y las peticiones a las rutas API.

Para construir la interfaz utilizamos React 19.2.4. React divide la web en componentes reutilizables y permite actualizar la pantalla cuando cambia el estado. Esto se aprecia, por ejemplo, al iniciar sesión, abrir el chat o guardar un favorito sin tener que recargar toda la página.

Sobre React utilizamos Next.js 16.2.0 como framework. Next.js organiza la aplicación mediante App Router: las páginas se encuentran dentro de la carpeta app y los endpoints del servidor se implementan como rutas API. De esta forma mantenemos la interfaz y la lógica del servidor dentro del mismo proyecto.

Para los estilos incluimos Tailwind CSS. En nuestro prototipo combinamos Tailwind con estilos en línea para ajustar rápidamente elementos concretos como tarjetas, formularios y modales.

La base de datos es PostgreSQL. Allí almacenamos clientes, empleados, vehículos, ventas y favoritos. Elegimos una base de datos relacional porque estas entidades están conectadas entre sí: una venta pertenece a un cliente, a un empleado y a un vehículo.

Para comunicarnos con PostgreSQL utilizamos Prisma 5.22. Es el ORM del proyecto: traduce las operaciones del código a consultas sobre la base de datos y facilita operaciones importantes como actualizar el stock y crear una venta dentro de una misma transacción.

El asistente virtual utiliza Gemini, concretamente el modelo gemini-2.5-flash-lite. Antes de enviar una consulta, nuestra ruta de chat carga el catálogo actual desde PostgreSQL para que la respuesta tenga en cuenta los vehículos, los precios y el stock disponibles.

El despliegue está preparado para Vercel, una plataforma especialmente integrada con Next.js. Para las imágenes de los vehículos utilizamos Vercel Blob como almacenamiento de archivos.

💡

Esta es la diapositiva más larga. Id señalando cada logo mientras lo explicáis. Si el tiempo aprieta, resumid Tailwind y JavaScript en una sola frase.

💡

Ejemplo: Cuando un cliente pulsa «Comprar», React recoge la acción. JavaScript envía la petición a una ruta API de Next.js. Prisma consulta PostgreSQL, registra la venta y reduce el stock. Después Next.js devuelve la respuesta y React actualiza lo que ve el usuario.

7

Diagrama Entidad-Relación

⏱ ~1 min 15 segDiapositiva 7 de 16

El núcleo del modelo de datos está formado por cinco entidades: Cliente, Empleado, Vehículo, Venta y Favorito.

Un cliente puede tener varias ventas y varios favoritos. Un empleado también puede gestionar varias ventas. Cada venta relaciona a un cliente, un empleado y un vehículo, e incluye cantidad, total, fecha y estado.

El vehículo almacena los datos necesarios para mostrar una ficha completa: marca, modelo, precio, stock, imágenes y características como año, combustible, potencia, transmisión o kilometraje.

La tabla Favorito relaciona cliente y vehículo. Además tiene una restricción única para impedir que el mismo cliente guarde dos veces el mismo coche.

💡

Ejemplo: Si el cliente guarda un vehículo que ya estaba en su lista, la operación utiliza un upsert: se mantiene un único favorito en lugar de duplicarlo.

9

Funcionalidades — Visitante y Cliente

⏱ ~1 min 35 segDiapositiva 9 de 16

Un visitante puede entrar al catálogo y buscar por marca o modelo. También puede abrir la ficha detallada de un vehículo y utilizar el formulario de contacto.

Cuando intenta comprar o guardar un favorito, la aplicación le solicita iniciar sesión como cliente. Así se reserva la navegación pública para la consulta y las operaciones personales para usuarios identificados.

El cliente registrado puede guardar favoritos, eliminarlos desde su perfil y consultar sus solicitudes de compra. Durante la compra puede personalizar opciones como seguro, garantía, entrega o subvención, y el sistema calcula el total mostrado.

Al confirmar, se crea una venta pendiente y se reduce el stock dentro de una transacción de base de datos.

💡

Ejemplo para demo: Buscar un modelo, guardarlo como favorito, entrar en el perfil y comprobar que aparece allí. Después, al solicitar la compra, el stock se actualiza para evitar vender unidades que ya no están disponibles.

10

Funcionalidades — Panel Trabajador y Administrador

⏱ ~1 min 25 segDiapositiva 10 de 16

El trabajador dispone de un panel centralizado con pestañas para clientes, vehículos y ventas.

Desde ese panel puede dar de alta o editar vehículos, añadir imágenes, modificar el stock y consultar el detalle de cada venta. Cuando procesa una solicitud, la venta pasa a estado realizada y el cliente recibe un correo de confirmación.

También puede cancelar una solicitud. En ese caso, la aplicación devuelve las unidades al stock y envía al cliente un correo informativo.

El administrador hereda estas funciones y añade la gestión de trabajadores: puede crearlos, editarlos, eliminarlos y asignar el rol correspondiente.

💡

Ejemplo para demo: Procesar una venta pendiente: se abre el detalle, se pulsa Procesar y el estado cambia a realizada. Si se cancela, la operación incrementa de nuevo el stock del vehículo.

11

Asistente Virtual con IA

⏱ ~1 min 35 segDiapositiva 11 de 16

El asistente virtual aparece como un widget flotante en toda la aplicación. El usuario puede abrirlo sin abandonar la página en la que está navegando.

La parte importante es que el asistente no trabaja con un catálogo fijo escrito a mano. Cada vez que recibe una consulta, la ruta del servidor recupera los vehículos actuales desde la base de datos y genera un contexto con marca, modelo, precio y stock.

Después se envía ese contexto junto con el historial de conversación a Gemini. También se limita el comportamiento mediante instrucciones: debe responder en español, ser conciso y centrarse en coches, precios, disponibilidad, comparativas y proceso de compra.

⭐

💡

Pregunta sugerida para demo: «Tengo un presupuesto de 20.000 euros, ¿qué vehículos disponibles me recomiendas?». La respuesta se basa en el inventario existente en ese momento.

12

Seguridad — Protección de datos

⏱ ~60 segDiapositiva 12 de 16

En protección de datos, las contraseñas de clientes y de nuevos empleados no se guardan directamente: se transforman con bcrypt antes de almacenarlas.

Las consultas generales de clientes excluyen la contraseña de la respuesta. Además, la recuperación de contraseña responde de la misma forma exista o no el correo, reduciendo la posibilidad de revelar qué usuarios están registrados.

La subida de imágenes valida que el archivo sea de tipo imagen. Para las imágenes almacenadas en Vercel Blob, el proxy admite únicamente direcciones HTTPS del dominio esperado.

💡

Ejemplo: Si alguien intenta registrarse con un correo ya utilizado, la aplicación devuelve un conflicto controlado en lugar de crear una cuenta duplicada.

👆

Señalad la columna password de la tabla mientras explicáis que son hashes, no contraseñas reales.

➡️

Transición: A esa protección añadimos una verificación adicional durante el acceso.

13

Seguridad — Verificación de acceso

⏱ ~60 segDiapositiva 13 de 16

Después de comprobar la contraseña, la aplicación envía un código numérico de seis cifras por correo electrónico. El usuario debe introducirlo para completar el acceso.

Cuando la verificación es correcta, el servidor genera una cookie firmada y httpOnly que permite confiar temporalmente en esa verificación. La firma impide modificar su contenido sin que el servidor lo detecte.

También existe un flujo de recuperación de contraseña mediante código enviado por correo y posterior almacenamiento de la nueva contraseña cifrada.

💡

Ejemplo para demo: Iniciar sesión, enseñar la pantalla de código y explicar que la cookie de confianza evita repetir la validación continuamente durante una sesión reciente.

Seguridad — Control de acceso por roles

⏱ ~60 segDiapositiva 14 de 16

La interfaz diferencia cliente, trabajador y administrador. El cliente se redirige a su perfil; el trabajador accede al panel operativo; y únicamente el administrador ve la pestaña de gestión de trabajadores.

Durante el desarrollo hemos comprobado manualmente flujos como registro, login con código, consulta del catálogo, favoritos, compra, actualización de stock y procesamiento de ventas.

Como siguiente refuerzo técnico, planteamos trasladar todas las comprobaciones de autorización a una sesión validada en servidor y añadir una batería de pruebas automatizadas. En un prototipo académico, esta evolución es importante para acercarlo a producción.

Preguntas previsibles del tribunal

Respuestas breves · ampliables si piden detalle
¿Por qué habéis utilizado Next.js?
Porque permite organizar en un mismo proyecto la interfaz con React y las rutas API del servidor. Para un TFG facilita mantener un flujo completo: página, lógica de servidor y acceso a datos.
¿Qué aporta realmente la IA?
No sustituye el catálogo. Lo complementa: recupera los vehículos actuales de PostgreSQL y ayuda a interpretar esa información mediante recomendaciones, comparativas y respuestas rápidas.
¿Cómo evitáis vender un coche sin stock?
Antes de crear la venta se comprueba el stock. Si es suficiente, la creación de la venta y la reducción de unidades se ejecutan juntas dentro de una transacción de Prisma.
¿Qué ocurre al cancelar una venta?
La venta cambia a estado cancelada, el stock se incrementa de nuevo y se envía un correo informativo al cliente.
¿Qué mejoraríais antes de llevarlo a producción?
Reforzaríamos la autorización en servidor para cada endpoint, usaríamos sesiones server-side, añadiríamos expiración persistente para los códigos, aplicaríamos validaciones más exhaustivas y crearíamos tests automatizados.
¿Qué pruebas habéis realizado?
En el prototipo se han revisado manualmente los flujos principales: registro, login y código, catálogo, favoritos, compra, stock y procesamiento de ventas. La automatización de tests queda como una mejora necesaria.
¿Vuestra arquitectura es MVC?
Guarda similitudes con MVC — podríamos equiparar Prisma al Model, React a la View y las API Routes al Controller — pero al estar el frontend y el backend desacoplados se describe mejor como arquitectura de 3 capas Cliente-Servidor. En MVC clásico el controlador devuelve la vista directamente desde el servidor; en nuestro caso React vive en el navegador y se comunica con las API Routes mediante fetch, son dos mundos separados.
¿Por qué una sola página muestra todos los vehículos con datos distintos?
Usamos enrutamiento dinámico de Next.js. El archivo /vehiculos/[id]/page.js actúa como plantilla: Next.js captura el ID de la URL, React hace una petición a la API Route con ese ID, y Prisma devuelve los datos de ese vehículo concreto desde PostgreSQL. Con 100 coches, si creáramos una página por vehículo tendríamos 100 archivos HTML. Así tenemos uno solo que se rellena dinámicamente.
¿Qué es Prisma y por qué lo habéis usado?
Prisma es un ORM (Object-Relational Mapper): una capa que traduce operaciones del código JavaScript a consultas SQL sobre PostgreSQL. Lo hemos usado porque permite definir el esquema de la base de datos en un archivo tipado (schema.prisma), aplica migraciones de forma controlada y facilita operaciones críticas como actualizar el stock y crear la venta dentro de la misma transacción, garantizando que nunca quede un estado inconsistente.
¿Qué es bcrypt y por qué las contraseñas no se guardan en texto plano?
bcrypt es un algoritmo de hashing diseñado para contraseñas. Transforma la contraseña en un hash irreversible, de modo que aunque alguien accediera a la base de datos no podría recuperar la contraseña original. Además añade un factor de coste (salt rounds) que hace el proceso deliberadamente lento para dificultar ataques de fuerza bruta.
¿Por qué habéis elegido PostgreSQL y no MySQL u otra base de datos?
PostgreSQL es una base de datos relacional robusta, de código abierto y con soporte nativo para transacciones ACID, que necesitamos para la lógica de stock y ventas. Además, Neon ofrece PostgreSQL gestionado en la nube con backups automáticos y se integra directamente con Vercel, que es nuestra plataforma de despliegue. La combinación Prisma + Neon + Vercel es el stack recomendado para proyectos Next.js modernos.
¿Qué diferencia hay entre un cliente registrado y un visitante?
El visitante puede navegar por el catálogo completo, ver fichas de vehículos y usar el asistente virtual. Cuando intenta comprar o guardar un favorito, la aplicación le redirige al login. El cliente registrado añade funciones personales: guardar y eliminar favoritos, solicitar compras, consultar su historial y gestionar su perfil. El acceso se diferencia mediante la cookie de sesión que genera el servidor al verificar el código de doble factor.
